Make orbital dynamics “mission-real” inside Flight Software.

OrbitSim is a compact F´ demo that turns orbital mechanics into a flight-software workflow: deterministic timing, command/telemetry loops, and mission-ops style observability. It is designed for teaching and for demoing how students’ equations become operational behavior inside embedded flight software.

Why this matters

In spacecraft and GSE-like workflows, “physics” is only half the story. The other half is how flight software makes those computations observable, testable, and safe under timing constraints.

  • Deterministic scheduler loop → repeatable timing and consistent behavior
  • Command-driven operation → “ops mindset” instead of ad-hoc scripts
  • Telemetry + events → traceable state changes and easy fault isolation
  • Hardware-ready demo path → easy to integrate with GPIO/LED indicators

What you can show in 60 seconds

1) Set scenario / mode (orbit propagation, state outputs) 2) Issue a command (change parameters, start/stop loop) 3) Observe telemetry trends (altitude, velocity, error metrics) 4) Point to deterministic scheduler + event log

Pipeline / architecture

OrbitSim follows a flight-software pattern: scheduled compute → state update → publish telemetry → log events. This keeps the demo “mission-real” and easy to extend to hardware I/O.

High-level block diagram (conceptual)

Scheduler (Svc.Sched) ↓ Orbit propagator (state update) ↓ Telemetry publish (position/velocity/errors) ↓ Event log (mode changes, command execution, anomalies) ↓ (Optional) hardware status out (LED / GPIO)

Teaching angle

  • Students can connect orbital equations to real-time execution constraints (dt, stability, drift).
  • Telemetry makes “what the code thinks” visible; events make “why it changed” visible.
  • Pairs naturally with OrbitDSP (filters/noise/fault injection) for signal+systems integration.
